So this is a stand-alone unit.  It takes opitons from the command line, and runs one of the "SUITE"s.  Each suite encapsulates a strategy for something to find in a JavaScript file: a function call, or a function invocation.  The program is designed to run on all .js files in a directory, to help migrate our codebase in the project root, infinite-neck/ which is written in old-school javascript, and is being migrated to ES6.

It produces adorned output, and also raw, organized output like grep.

The code runs, and produces the expected outputs: lists of global namespace javascript invocations in source files, and lists of exported and non-exported functions in the source files.

It also produces a structured runtime output array of objects that encapsulate state about the lines found in every source file.

So I'd like to drill down into the strategy for actually doing the replace.  In our case, the "First Pass" through the program (find-js-dependencies.js) has produced two structures:
1) an array of Line objects for invocations, where a Line object has these properties:
     a) a bare identifier
     b) the actual start index of the identifier exactly pointing to the code before any replacement.
     c) the entire line for context, which can be displayed to the software developer and even wrapped in html tags for colored display if needed. (Grabbing the entire line is not properly implemented in the code yet.)

2) a list of functions and exports that are candidates for the identifier based on what javascript file they came from.  These also have Line objects for displaying to the developer.  They don't need to be reference by the program on the "Second Pass" where replacement occurs, because the input to the "Second Pass" will just be a list of Namespace qualifiers for each invocation identifier.

So when the developer has reviewed the side-by-side or hierachically nested display of issues, he can choose the invocation identifiers and associate them with the Namespace qualifier.  This would be an input data structure to the "Second Pass" when the program is run in replacement mode.

So it seems that the replacement strategy is to rip through the source file with the invocations, one by one from the list of identifiers and their locations, and build a reference to source line and replaced line.

Then the process is repeated for the next identifier. Hopefully in the same run so we don't get into higher order search time inefficiencies.  The issue becomes how to build up the output, because every search and replace would change the source file if we did a brute force approach.  

Instead, it seems it would be better to insert a magic tag in the source that gets replaced as the new file is rebuilt.  The magic tag is effectively a reference to the replacement point, and all replacement points are accumulated and replaced on final output once.  This would be in a data structure, not an actual string.

We can simplify the use case by stating a rule:

   "No method invocations that are flagged for inclusion will span more than one line of source code."

This rule will be enforced by the developer by lint'ing the file before submission.

It's not a lexical parser, so this would violate the rule, but may work anyway:

    invocationOne(  "argument1",
        "argument2",
        argument3);
    invocation2();

This next examplle would violate the rule, and the developer would ensure that some other strategy is used by changing the source file first, because the code is messy anyway:

    invocationOne( `${invocationTwo()} baz baz`, 
                   invocationThree());

It's not clear if this would actually present a problem, but let's assume that we'll enforce such rules in developer best practices, not in code.

But lets assume that we will handle these cases correctly:

    invocationOne(invocationTwo());

because this becomes

We can also assume that the following rule will be lint'd by the developer:
